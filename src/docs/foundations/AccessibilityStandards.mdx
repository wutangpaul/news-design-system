# Accessibility Standards

**Target: WCAG 2.2 AA**, non-negotiable. This isn't a checklist applied at the end — it's a
build constraint every component is designed against from the start, because retrofitting
accessibility into a sealed component API is far more expensive than designing it in.

## Non-negotiables for every component

- **Keyboard operability.** Every interactive element is reachable and operable with Tab,
  Shift+Tab, Enter, Space, and arrow keys where the
  [WAI-ARIA Authoring Practices Guide](https://www.w3.org/WAI/ARIA/apg/) defines a pattern for
  it (Tabs, Accordion, Dropdown, Menu, Dialog). Components implement the APG pattern, not a
  bespoke interpretation of it.
- **Semantic HTML first.** Reach for the native element that already has the right behavior
  (`<button>`, `<select>`, `<input type="radio">`) before reaching for a `div` plus ARIA. Native
  elements give keyboard handling, focus management, and AT announcement for free; ARIA is for
  filling gaps native HTML can't cover, not a replacement for it.
- **Visible focus.** A global `:focus-visible` ring is defined once in `src/index.css` and
  inherited by every component automatically. No component suppresses it with `outline-none`
  without supplying an equally visible replacement.
- **Color is never the only signal.** Status (error/success/warning), state (selected/disabled),
  and meaning are always paired with text, an icon, or shape — see Alert and Badge for the
  reference pattern.
- **Labels are programmatic, not just visual.** Form fields associate their `<label>` via
  `htmlFor`/`id` (generated with `useId` when the consumer doesn't supply one); helper and
  error text are wired through `aria-describedby`, not just placed nearby visually.
- **Motion respects `prefers-reduced-motion`.** Spinner, Skeleton, and Progress all fall back to
  a static or non-spinning state — see Spacing & Motion in
  [Shadows & Motion](?path=/docs/design-tokens-shadows-motion--docs).
- **Focus management for overlays.** Modal and Drawer trap focus while open, move focus in on
  open, and return focus to the trigger on close, per the APG Dialog pattern.

## Verification

- Storybook's `addon-a11y` runs axe-core against every story automatically — a clean run is part
  of "done" for any component, not an optional follow-up.
- Component tests assert real DOM behavior (roles, `aria-*` attributes, keyboard interaction),
  not just that something rendered — see any `*.test.tsx` for the pattern.

## Semantic color contrast audit (2026-07-03)

Every `success`/`warning`/`error`/`info` text/background/icon combination actually used in the
codebase was checked against its real rendered background (computing WCAG relative luminance
directly from the token hex values, not eyeballed) in both light and dark mode. Two real
failures were found and fixed:

- **Toast's warning icon** (`text-warning-500` on `bg-surface-raised`) was 2.97:1 in light
  mode — just under the 3:1 AA floor for non-text/UI components (1.4.11). Fixed to
  `text-warning-700 dark:text-warning-500` (the 700 shade clears 4.68:1 in light and still
  passes 3.55:1 in dark).
- **The required-field asterisk** (`text-error-500`, used identically across Input, Select,
  Textarea, Checkbox, DatePicker, and Combobox) was 4.03:1 against `surface-canvas` in dark
  mode — under the 4.5:1 AA text floor. Fixed with `dark:text-error-400` (7.04:1). This is the
  same fix Button's destructive variant already used for the identical root cause: `error-500`
  was tuned for light backgrounds and doesn't have enough luminance contrast against near-black
  ones.
- Also removed **Form's error-summary link hover color** (`hover:text-error-500`), which
  reached only 4.41:1 in light mode and 3.79:1 in dark — both under 4.5:1. The link's
  persistent underline is affordance enough without a hover color change.

Everything else — Alert's tinted backgrounds/icons in both themes, Badge's fixed pastel
chips, the Alert dark-mode text-on-tinted-background pairs — passed with real margin (many
well above 10:1). Method + full numbers available via the contrast-checker script used for
this pass if another token changes and needs re-verification.
