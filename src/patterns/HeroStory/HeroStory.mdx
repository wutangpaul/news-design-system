# Hero Story

A large featured-story treatment for a homepage or section top: a full-bleed `Image`
with an overlaid category `Tag`, headline, and optional dek. Composed entirely from
existing components — `Image`, `Heading`, `Tag` — rather than hand-rolled markup.

<Canvas of={HeroStoryStories.Default} />
<Controls />

## Composition, not a link

Like `Card`, `HeroStory` never renders its own click handler, `role="button"`, or
anchor wrapper. Make the whole unit clickable by wrapping it in your own `<a>` or
router `Link`:

```tsx
import { HeroStory } from "@/patterns/HeroStory";

<a href="/story/123" className="block rounded-lg">
  <HeroStory
    category="Climate"
    title="Coastal cities race to finish sea walls before hurricane season"
    imageSrc="/images/sea-wall.jpg"
    imageAlt="Workers reinforcing a concrete sea wall at sunset"
  />
</a>;
```

<Canvas of={HeroStoryStories.ComposedAsALink} />

This keeps `HeroStory` framework-agnostic (no dependency on a particular router) and
keeps focus order / keyboard activation owned by the real anchor element.

## Optional category and dek

Both `category` and `dek` are optional — a hero with just an image and headline is a
common minimal case.

## Aspect ratio

`aspectRatio` is forwarded straight to `Image` — pick whichever of `Image`'s editorial
presets suits the layout slot (a square crop for a dense grid tile, `16/9` for a
full-width lead).

<Canvas of={HeroStoryStories.SquareAspectRatio} />

## Heading level

`headingLevel` defaults to `2`, since a hero on a homepage or section front usually
sits below the page's own `h1` (the site or section title). Set it to `1` when the
hero genuinely is the page's primary heading, e.g. a single-story special page.

<Canvas of={HeroStoryStories.HeadingLevelTwo} />

## Do's and don'ts

- Do wrap `HeroStory` in a real `<a>`/`Link` for the clickable case — never add a click
  handler or `role="button"` to the pattern itself or a wrapping `div`.
- Do set `headingLevel` to match where the hero actually sits in the page's document
  outline — don't leave the default `2` on a page where the hero is the sole `h1`.
- Don't put interactive controls (buttons, form fields) inside the overlay text area —
  the overlay is a caption for the image, not a UI panel, and doing so would create a
  larger tap target that's ambiguous about what it activates once the whole card is
  wrapped in a link.

## Accessibility

- The gradient scrim behind the overlay text (`bg-gradient-to-t from-ink-950/90 ...`)
  and the overlay text color (`text-ink-0`) use the raw `ink-*` scale directly rather
  than the theme-aware `surface-*`/`text-*` aliases — this is the "fixed-brand element"
  exception CONTRIBUTING calls out: the overlay sits on top of a photograph, not the
  app's canvas, so it must stay legible regardless of the light/dark theme toggle. A
  themed `text-text-primary` would go dark-on-dark-photo in dark mode.
- The scrim itself is `aria-hidden="true"` and purely decorative (a gradient div with no
  content) — only the real heading/tag/dek text is exposed to assistive tech.
- `Image`'s `alt` is required and forwarded as-is; always describe the photo's content,
  not the headline (the headline is already available as text).
